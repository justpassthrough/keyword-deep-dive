# 키워드 딥다이브 스캐너 — 작동 원리 & 수정 이력

## 1. 이 도구가 뭐하는 건가요?

약사 블로그에 쓸 글감을 자동으로 찾아주는 도구입니다.

**핵심 흐름:**
```
"마운자로" (뿌리 키워드)
    → "마운자로 부작용", "마운자로 품절", "마운자로 감량" ... (복합 키워드 자동 확장)
    → 각각에 대해 검색량, 변화율, 전문가 갭, 약사 가치 점수를 매김
    → 대시보드에 정리해서 보여줌
```

**대시보드**: https://justpassthrough.github.io/keyword-deep-dive/

---

## 2. 파일 구조

```
keyword-deep-dive/
├── scripts/
│   ├── dive.py              ← 핵심. 복합키워드 확장 + 분석
│   ├── build_dashboard.py   ← latest.json → HTML 대시보드
│   └── discover_roots.py    ← 새 뿌리 키워드 자동 발굴
├── data/
│   ├── root_keywords.json   ← 뿌리 키워드 목록 (현재 13개)
│   ├── known_products.json  ← 제품/성분 이름 사전 (40개)
│   ├── latest.json          ← 최근 분석 결과
│   └── history/             ← 과거 분석 결과 누적
├── docs/
│   └── index.html           ← GitHub Pages 대시보드
├── .github/workflows/
│   ├── daily_dive.yml       ← 하루 4회 자동 실행
│   └── weekly_discover.yml  ← 3일마다 뿌리 발굴
└── requirements.txt
```

## 4. 각 스크립트가 하는 일

### dive.py (메인 분석)

뿌리 키워드 하나마다 이 과정을 반복합니다:

1. **복합키워드 마이닝**: 네이버 블로그/뉴스 제목 100개씩 가져와서, "뿌리+X" 패턴의 복합키워드 추출
2. **SEO 스팸 필터**: 지역명/숫자를 마스킹해서 같은 패턴 3건 이상이면 스팸으로 판단, 제거
3. **DataLab 비교**: 네이버 DataLab에서 검색량 조회 (4개씩 배치, 1초 간격)
4. **변화율 계산** (듀얼 시스템):
   - **change_rate** (메인): 최근 3일 vs 직전 3일 → 빠른 급등 감지
   - **trend_rate** (보조): 최근 7일 vs 이전 7일 → 전체 추세 참고
5. **롱테일 판단**: DataLab에 안 잡히는 키워드 → 블로그 totalCount로 수요 판단
6. **전문가 갭**: 전체 블로그 수 / (약사 블로그 수 + 1) 비율 계산
7. **약사 가치 산출**: `의도점수 × 전문가갭배수 × 변화율보너스`
8. **미확인 후보**: known_products 사전에 없는 빈도 높은 단어 수집
9. **추이 비교**: 이전 실행 결과와 비교해서 새 급등/새 복합키워드/변화율 변동 감지
10. **생명주기 업데이트**: active/watch/dormant 상태 전환
11. **API 사용량 추적**: DataLab 호출 수 카운트, 일일 한도 80% 초과 시 경고

### build_dashboard.py (대시보드)

latest.json을 읽어서 HTML 대시보드를 생성합니다:
- 추천 글감 TOP 3 (약사가치 최고순)
- 이전 대비 변화 섹션 (새 급등, 새 복합키워드, 변화율 상승/하락)
- 뿌리별 탭 (active + watch 모두 동일하게 표시, watch는 👀 표시)
- 미확인 후보 섹션
- API 한도 경고 배너 (80% 초과 시)

### discover_roots.py (뿌리 발굴)

4가지 소스에서 새로운 뿌리 키워드를 찾습니다:
- **소스 A**: active 뿌리 블로그에서 known_products 사전에 있는 다른 제품명 발견
- **소스 B**: "비만치료제 신약", "영양제 인기 순위" 등 카테고리 시드 쿼리
- **소스 C**: "의약품 허가", "건강기능식품 출시" 등 약업계 뉴스
- **소스 D**: health-trend-scanner의 latest.json에서 제품명 추출

후보 검증: DataLab 검색량 있음 + 복합키워드 5종 이상 → `watch`로 등록

## 5. 대시보드 읽는 법

### 약사 가치 점수
```
약사가치 = 의도점수(1~4) × 전문가갭배수(0.7~1.3) × 변화율보너스(1.0~1.4)
```
**고정값이 아닙니다** — 전문가갭과 변화율은 매 실행마다 바뀌므로 점수도 변동됩니다.

### 라벨
| 라벨 | 의미 | 예시 |
|------|------|------|
| ★★★★ | 약사 전문성 필수 (4점) | 부작용, 상호작용, 복용법, 성분, 비교 |
| ★★★ | 약사 관점 유용 (3점) | 효과, 품절, 대체, 추천 |
| ★★ | 경험담 위주 (2점) | 후기, 전후, 결과 |
| ★ | 구매/접근성 (1점) | 가격, 최저가, 약국 |
| 🔥급등 | 3일 변화율 +20% 이상 | |
| 🔗비교형 | 다른 뿌리 키워드 포함 | "마운자로 위고비 차이" |
| 📎롱테일 | DataLab에 없지만 블로그 수요 있음 | |

### 전문가 갭
- **전문가 갭 큼**: 전체블로그/(약사블로그+1) 비율 30배↑ → 글 쓰기 좋은 기회
- **전문가 부족**: 비율 10배↑
- **보통**: 비율 3배↑
- **전문가 포화**: 비율 3배 미만 → 경쟁 심함

### 검색량
- **숫자 (예: 17.6)**: DataLab 상대 검색량 (뿌리 대비 비율)
- **블로그 230**: DataLab에 안 잡히는 롱테일, 블로그 검색 수로 표시

### 변화율
- **메인 숫자**: 3일 단기 변화율 (최근 3일 vs 직전 3일)
- **아래 회색 글씨 "7일 +X%"**: 7일 추세 변화율 (보조 참고용)

## 6. 뿌리 키워드 생명주기

```
watch → (복합키워드 5개↑) → active → (28일 급등 0) → dormant → (56일) → archived
                                ↑                          │
                                └──(급등 재등장)────────────┘
```

- **날짜 기반 계산**: last_active로부터 경과일 ÷ 7 = 주수. 하루 4회 실행해도 중복 카운트 안 됨
- **watch**: 분석은 active와 동일하게 수행. 대시보드에서도 동일한 탭+테이블로 표시 (👀 구분)
- **현실적으로 active → dormant는 잘 안 일어남**: 복합키워드가 수십 개인데 28일간 하나도 +20% 안 뜰 확률 낮음. 뿌리가 많아지면 그때 기준 강화 예정

## 7. API 한도 관리

| API | 일일 한도 | 병목 여부 |
|-----|----------|----------|
| DataLab | 1,000회 | **병목** |
| 블로그/뉴스 검색 | 25,000회 | 여유 |

**뿌리당 DataLab 약 5~6회/실행** → 현재 13개 뿌리 × 4회/일 ≈ 312회 (31%)

**경고 시스템**: dive.py가 매 실행마다 DataLab 호출 수를 세서, 하루 4회 기준 80% (800회) 초과 예상 시:
- 로그에 경고 출력
- 대시보드 최상단에 빨간 경고 배너 표시

**한도 도달 예상: 뿌리 약 40개** → 그때 dormant 기준 강화하거나 약사가치 낮은 뿌리 정리

## 8. 미확인 후보 필터링

블로그 제목에서 추출된 단어 중 known_products 사전에 없는 것들을 걸러내는 다중 필터:

1. **의도 키워드 제외**: 부작용, 효과, 후기, 가격 등 (HIGH/MID/LOW/WEAK_PHARMA 단어들)
2. **CANDIDATE_EXCLUDE**: 카테고리어(비만약, 영양제), 마케팅수식어(프리미엄, 골드), 기업명(대원제약, 종근당), 일반건강용어(건강, 체중, 단백질)
3. **뿌리 합성어 제외**: "위고비마운자로" 같은 뿌리끼리 합쳐진 단어
4. **조사 처리**: "킴스클럽과" → "킴스클럽" 매칭 (은/는/이/가/을/를/과/와/의 등 제거)

## 12. 알려진 한계 & 향후 개선 가능

- **discover_roots.py는 known_products 사전에 있는 이름만 뿌리로 등록**: 완전히 새로운 제품/성분은 미확인 후보로만 노출 → 사람이 보고 사전에 추가해야 함
- **dormant 강등이 현실적으로 잘 안 일어남**: 뿌리가 많아지면 기준 강화 필요 (예: 급등 비율 10% 이상 등)
- **카테고리 자동 추정이 부정확할 수 있음**: 유산균이 "비만치료제"로 분류된 사례. 수동 수정 가능
- **git 충돌**: Actions 자동 커밋과 로컬 수정이 겹치면 충돌. `git pull --rebase` 습관화 필요
